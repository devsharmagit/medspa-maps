# MedSpa Maps — Database Reference

> **Purpose:** Authoritative schema reference for the MedSpa Maps application database, generated by direct introspection of the live Postgres instance. Use this in future conversations instead of re-scraping the DB.
>
> **Generated:** 2026-07-08 · **Source:** live introspection (`information_schema` + `pg_catalog`)
> **Engine:** PostgreSQL 18.4 · **PostGIS** 3.6.0
> **Schema:** `public` · **21 base tables + 1 materialized view**

Connection is via `DATABASE_URL` (pooler, `sslmode=require&channel_binding=require`). This is the **application DB** — distinct from the read-only **G99 prod Aurora DB** (accessed via SSH tunnel on `:5434`, documented separately; the `g99_*` tables below are a *harvested copy* of G99 website data, not a live link).

## 2. How everything connects (entity map)

```
                         ┌───────────────┐
                         │  businesses   │  (tenant / brand)
                         └──────┬────────┘
                    RESTRICT    │ 1        └───────< listing_claims (CASCADE)
                                │
                                ▼ *
                         ┌───────────────┐
       ┌─────────────────│    clinics    │──────────────────────────┐
       │ CASCADE         └──────┬────────┘  CASCADE                  │ CASCADE
       │                        │                                    │
       ▼ *                      ▼ *                                  ▼ *
 clinic_locations         clinic_services ──SET NULL──> services   providers
 (addresses + geo)              │  (join w/ price)          ▲          │ CASCADE
                                │                           │          ▼ *
       ┌────────────────────────┘             concern_services (M:N)  provider_services ──> services (M:N)
       │                                        │        ▲            provider_concerns ──> concerns (M:N)
       ▼ *                                      ▼        │
 clinic_concerns ──CASCADE──> concerns <────────┘        │
                                  ▲                       │
 reviews ──> clinics (CASCADE), ──┘ concerns/services (SET NULL), provider_id (no FK)
 scrape_jobs ──CASCADE──> clinics
 clinic_service_changes ──> clinics (CASCADE), services & scrape_jobs (SET NULL)
 images  ── POLYMORPHIC (entity_type + entity_id, NO FK) ── clinic | business | ...

 Standalone (no FKs): admin_users · medspa_leads · postal_codes · g99_clinic_websites
```

**The spine:** `businesses` → `clinics` → (`clinic_locations`, `clinic_services`, `providers`, `reviews`, `images`, `scrape_jobs`). A **business** is the tenant/brand; a **clinic** is a listing under it; a clinic has many physical **locations**, offers many **services**, employs **providers**, and accrues **reviews**/**images**.

**Two catalogs (locked taxonomy — see §5):** `services` (15 canonical treatments) and `concerns` (10 conditions), linked M:N via `concern_services`. Clinics/providers attach to them via `clinic_services`/`clinic_concerns`/`provider_services`/`provider_concerns`.

**Foreign-key delete rules (all 21 FKs):**

| Child → Parent | On delete |
|---|---|
| `clinics.business_id` → `businesses.id` | **RESTRICT** (can't delete a business with clinics) |
| `clinic_locations.clinic_id` → `clinics.id` | CASCADE |
| `clinic_services.clinic_id` → `clinics.id` | CASCADE |
| `clinic_services.service_id` → `services.id` | SET NULL |
| `clinic_concerns.clinic_id` / `.concern_id` | CASCADE / CASCADE |
| `providers.clinic_id` → `clinics.id` | CASCADE |
| `provider_services.*` / `provider_concerns.*` | CASCADE (both sides) |
| `concern_services.concern_id` / `.service_id` | CASCADE / CASCADE |
| `reviews.clinic_id` → `clinics.id` | CASCADE |
| `reviews.service_id` / `.concern_id` | SET NULL / SET NULL |
| `scrape_jobs.clinic_id` → `clinics.id` | CASCADE |
| `clinic_service_changes.clinic_id` → `clinics.id` | CASCADE |
| `clinic_service_changes.service_id` / `.scrape_job_id` | SET NULL / SET NULL |
| `listing_claims.business_id` → `businesses.id` | CASCADE |

> ⚠️ **Gotchas:** (1) `images` is **polymorphic** — `entity_type`+`entity_id` with **no FK**, so referential integrity is app-enforced. (2) `reviews.provider_id` is a `uuid` column with **no FK constraint** (unlike its clinic/service/concern siblings). (3) `services.is_published` is effectively dead per project memory — use `is_active`.

## 7. `clinic_search_view` (materialized view — search backbone)

Powers the search API. **Unique index on `clinic_id`** ⇒ supports `REFRESH MATERIALIZED VIEW CONCURRENTLY`.

**Shape:** one row per active clinic (`clinics.is_active AND businesses.is_active`), joined to its business, with services aggregated into arrays and cover/logo images resolved via correlated subqueries.

**Key derived columns:**
- `service_names` `text[]` — `COALESCE(services.name, clinic_services.raw_name)` for active services.
- `service_slugs` `text[]` — `COALESCE(services.slug, slugify(raw_name))`; **GIN indexed** for treatment filtering.
- `cover_image_url` — first `images` row where `entity_type='clinic' AND role='cover' AND scrape_status='ok'`.
- `logo_url` — first `images` row where `entity_type='business' AND role='logo' AND scrape_status='ok'`.
- Plus flattened clinic/business/geo/contact/rating columns.

**Indexes:** unique `clinic_id`; GiST `geo`; GIN `service_slugs`; btree `lower(city)`, `lower(state)`, `avg_rating DESC`, `tier`.

> Refresh after any ingest/rescrape that changes clinics, services, or images. Excludes inactive clinics/businesses, so deactivating hides a clinic from search on next refresh.

## 8. Notable indexing patterns

- **Geo search:** GiST on `clinics.geo`, `clinic_locations.geo` (partial), `clinic_search_view.geo`.
- **Full-text:** GIN `to_tsvector` on `clinics(name,city)` and `clinic_services(raw_name,description)`.
- **Autocomplete:** trigram GIN on `postal_codes.place_name` + zip-prefix btree.
- **Taxonomy filter:** GIN on `clinic_search_view.service_slugs`.
- **Visibility/soft-delete:** `is_active` btree indexes on businesses, clinics, clinic_services, providers.
- Every FK column is indexed; hot lookup columns (`slug`, `website`, `google_place_id`, `g99_*_id`, status/tier/role) indexed.
