# MedSpa Maps — Database Reference

> **Purpose:** Authoritative schema reference for the MedSpa Maps application database, generated by direct introspection of the live Postgres instance. Use this in future conversations instead of re-scraping the DB.
>
> **Generated:** 2026-07-18 (after the schema simplification — see the migration `web/scripts/2026-07-18-simplify-schema.sql`) · **Source:** live introspection (`information_schema` + `pg_catalog`)
> **Engine:** PostgreSQL 18.4 · **PostGIS** 3.6.0
> **Schema:** `public` · **15 base tables + 1 materialized view** (+ PostGIS system table `spatial_ref_sys`, not counted above)

Connection is via `DATABASE_URL` (Neon pooler, `sslmode=require&channel_binding=require`). This is the **application DB** — distinct from the read-only **G99 prod Aurora DB** (accessed via SSH tunnel, documented separately; the `g99_clinic_websites` table below is a *harvested copy* of G99 website URLs, not a live link).

> ⚠️ **pg client version:** the DB is Postgres 18; the default Homebrew `pg_dump`/`psql` is 16 and **refuses on version mismatch**. Use the pg18 binaries at `/opt/homebrew/Cellar/postgresql@18/*/bin/`.

---

## 0. What changed in the 2026-07-18 simplification

The DB was deliberately reduced to **clinic basic details + treatments + concerns + the clinic/treatment/concern joins**. Removed:
- **Tables dropped:** `businesses` (dissolved into `clinics`), `listing_claims`, `medspa_leads`, `scrape_jobs`, `clinic_service_changes`, `concern_services`, `clinic_concern_evidence`.
- **`businesses` → `clinics`:** `clinics.business_id` removed; `g99_business_id` + `g99_tenant_id` now live on `clinics`; `clinics.slug` is now **globally unique** (was unique per business).
- **`clinics` columns dropped:** city, state, zip, geo, lat, lng, tier, verified, founded_year, stat_* — addresses/geo now live **only** in `clinic_locations`.
- **`services` / `concerns`** reduced to bare search catalogs (id, name, slug, is_active, origin).
- **`clinic_services`** lost match_confidence, data_source, scraped_from_url, last_scraped_at.
- **`providers`** reduced to card fields (name, title, image_url, is_verified, card_tagline).
- **Concern search no longer requires evidence** — membership in `clinic_concerns` is sufficient (the evidence-quote gate + `clinic_concern_evidence` table are gone).

## 2. How everything connects (entity map)

```
                    ┌───────────────┐
        ┌───────────│    clinics    │───────────────────────────┐
        │ CASCADE   └──────┬────────┘  CASCADE                   │ CASCADE
        ▼ *                │                                     ▼ *
  clinic_locations         ▼ *                                providers
  (addresses + geo)   clinic_services ──SET NULL──> services
                           │  (join w/ price)          ▲
        ┌──────────────────┘                           │
        ▼ *                                             │
  clinic_concerns ──CASCADE──> concerns <──────┐        │
                                     ▲          │        │
              clinic_service_concerns ──────────┴────────┘
              (clinic + service + concern; all 3 FKs CASCADE)

  reviews ──> clinics (CASCADE), services/concerns (SET NULL), provider_id (no FK)
  images  ── POLYMORPHIC (entity_type + entity_id, NO FK) ── entity_type='clinic'
  ai_navigator_events ──SET NULL──> ai_navigator_sessions

  Standalone (no FKs): admin_users · postal_codes · g99_clinic_websites
```

**The spine:** a **clinic** is the top-level listing (no more `businesses` tenant). A clinic has many physical **locations** (`clinic_locations` — the sole home of addresses/geo now), offers many **services** (`clinic_services`), links concerns it treats (`clinic_concerns`), employs **providers**, and accrues **reviews**/**images**. `clinic_service_concerns` is the single per-clinic **clinic↔treatment↔concern** link (written by ingest; see note below).

**Two catalogs (curated seed + AI-grown):** `services` (15 `origin='seed'` + 286 `origin='ai'` = 301) and `concerns` (10 `origin='seed'` + 118 `origin='ai'` = 128). Both are now **bare** — name/slug/is_active/origin only (no descriptions, aliases, prices, or SEO copy).

**Foreign keys (13 total):**

| Child → Parent | On delete |
|---|---|
| `clinic_locations.clinic_id` → `clinics.id` | CASCADE |
| `clinic_services.clinic_id` → `clinics.id` | CASCADE |
| `clinic_services.service_id` → `services.id` | SET NULL |
| `clinic_concerns.clinic_id` / `.concern_id` → clinics / concerns | CASCADE / CASCADE |
| `clinic_service_concerns.clinic_id` / `.service_id` / `.concern_id` | CASCADE / CASCADE / CASCADE |
| `providers.clinic_id` → `clinics.id` | CASCADE |
| `reviews.clinic_id` → `clinics.id` | CASCADE |
| `reviews.service_id` / `.concern_id` | SET NULL / SET NULL |
| `ai_navigator_events.session_id` → `ai_navigator_sessions.id` | SET NULL |

> ⚠️ **Gotchas:** (1) `images` is **polymorphic** (`entity_type`+`entity_id`, no FK) — all rows are `entity_type='clinic'` now (business-logo rows are gone). (2) `reviews.provider_id` has **no FK**. (3) `clinic_service_concerns` is **written by ingest but not yet read by any app query** — it's the intended clinic-treatment-concern link but currently unused for display/search. (4) `clinic_services` lost `data_source`, so the rescrape cron can no longer distinguish manual vs scraped rows — it deactivates any active row not re-detected. (5) `origin` (`seed`/`ai`/`manual`) marks curated-vs-AI-grown catalog rows; `reconcile-taxonomy.ts` only deletes `origin='seed'` rows.

## 7. `clinic_search_view` (materialized view)

> ⚠️ **Not read by any app query** — search reads the base tables directly. The matview is only `REFRESH`ed (kept valid so existing refresh call sites work). If you want to retire it, remove the refresh calls in `src/app/api/admin/clinics/save/route.ts`, `src/app/api/internal/rescrape/refresh-view/route.ts`, `src/lib/admin/website-import.ts`, `src/lib/admin/queue.ts`, `src/app/api/admin/unmatched/map/route.ts`.

**Shape:** one row per active clinic (`c.is_active`), services aggregated into arrays, cover/logo resolved via correlated subqueries. Columns: `clinic_id, clinic_name, clinic_slug, website, booking_url, about, phone, avg_rating, review_count, ext_rating, ext_review_count, featured, service_names text[], service_slugs text[], cover_image_url, logo_url`.

**Indexes:** unique `idx_csv_clinic_id` (enables `REFRESH … CONCURRENTLY`); GIN `idx_csv_service_slugs`.

## 9. Notable indexing patterns

- **Geo search:** GiST on `clinic_locations.geo` (partial) — the only geo column now.
- **Full-text:** GIN `to_tsvector` on `clinic_services(raw_name, description)`.
- **Autocomplete:** trigram GIN on `postal_codes.place_name`.
- **Taxonomy filter:** GIN on `clinic_search_view.service_slugs`; `idx_services_slug`, `idx_concerns_slug`.
- **Visibility:** `is_active` btree on clinics, clinic_services, providers.
- Every FK column is indexed; `clinics.slug`/`website`/`google_place_id`/`g99_clinic_id` indexed.

## 10. How to regenerate

```bash
export PGURL="$DATABASE_URL"
PGDUMP=/opt/homebrew/Cellar/postgresql@18/*/bin/pg_dump   # pg18 — default brew pg16 refuses (version mismatch)

cp web/db/schema.sql web/scripts/schema.sql   # keep the duplicate in sync
# Then refresh this doc from information_schema + pg_catalog (see the queries used to build §3–§9).
```
